# LibreCell 工程化标准单元库自动生成平台 - 需求规格说明书

## 1. 项目概述

基于 LibreCell 开源项目（版本 0.0.9~0.0.11，Alpha 阶段），构建一个**可工程化落地**的 CMOS/BCD 标准单元库自动生成平台。

### 1.1 当前 LibreCell 能力分析

| 能力维度 | 当前状态 | 评估 |
|---------|---------|------|
| 布局生成 | SPICE → Placement → Routing → GDS/LEF/MAG | 基本可用，但硬编码严重 |
| 单元类型 | 组合逻辑 + 时序单元（DFF/Latch） | 覆盖主流标准单元 |
| 布局算法 | Euler tour / Hierarchical / SMT / SA 等 6 种放置器 + PathFinder 布线 | 算法丰富，可扩展 |
| 时序表征 | ngspice 仿真 → NLDM Liberty 文件 | 基本可用 |
| 工艺配置 | Python 文件，无 schema 校验 | 可用但脆弱 |
| 测试体系 | 无 pytest 框架，仅有零散内联测试 | **严重不足** |
| CI/CD | 无 | **缺失** |
| 文档 | 仅 README | **不足** |

### 1.2 关键瓶颈与阻塞问题

1. **硬编码层定义**：`layers.py` 仅定义 ndiff/pdiff/nwell/pwell/poly/metal1/metal2/via1，仅支持 2 层金属
2. **双行结构固化**：`Cell` 数据类型写死 `upper`/`lower` 两行，不支持多行单元
3. **单电源域**：断言 exactly 1 VDD + 1 GND，不支持 BCD 多电压域
4. **固定栅长**：全局 `gate_length`，不支持多 Vt
5. **无阱接触（Well Tap）**：LVS 使用 3 端子 MOSFET
6. **Common 包反向依赖 Layout**：`lccommon.net_util` 导入 `lclayout.data_types`
7. **依赖版本过旧且锁定过严**：`klayout==0.26.*`, `sympy==1.6.*`, `pyspice==1.4.*`
8. **无形式化测试**：零 pytest 测试，无法保证重构安全性

### 1.3 核心原则

- **先测试，后重构**：每一阶段先补测试，确保改动不破坏现有功能
- **渐进式改造**：不做大爆炸式重写，每阶段可独立验证
- **配置驱动**：将所有硬编码提取为配置，核心算法不变
- **向后兼容**：现有 `dummy_tech.py` 等示例在重构后仍能正常工作

## 2. 最终目标

| 编号 | 目标 | 优先级 |
|-----|------|--------|
| G1 | CMOS 标准单元库生成（180nm/90nm/55nm/40nm/28nm） | P0 |
| G2 | BCD 工艺标准单元库生成 | P1 |
| G3 | 工艺配置可扩展（不同 foundry / 不同节点） | P0 |
| G4 | 同工艺不同 track 迁移（如 6T → 7T, 9T -> 7T, 7T->6T, 10T -> 7T等等） | P1 |
| G5 | 跨工艺节点迁移（如 90nm → 55nm） | P1 |
| G6 | 脚本化二次开发（SE 工程师仅通过配置 + Python 脚本适配） | P0 |
| G7 | 架构可扩展（未来支持 FinFET，当前不实现） | P2 |

### 3.2 关键设计决策

| 决策项 | 方案 | 理由 |
|-------|------|------|
| 工艺配置格式 | YAML（主）+ Python（高级扩展） | YAML 对 SE 友好，Python 保留灵活性 |
| Schema 校验 | Pydantic v2 数据模型 | 类型安全 + 自动文档 + 默认值 |
| 层定义 | 运行时从配置构建 LayerStack | 消除硬编码，支持任意金属层数 |
| 多电源域 | TechConfig 中声明 power_domains 列表 | 支持 BCD HV/LV 混合 |
| 多 track | 参数化 unit_cell_height / routing_grid_pitch | 同一工艺 7T/9T 仅改配置 |
| 包结构 | 将 data_types 移入 librecell-common | 解除循环依赖 |

## 4. 实施阶段总览

共分 **10 个阶段**，严格按顺序执行。每阶段有明确的验收标准（测试用例通过），上一阶段完成才进入下一阶段。

| 阶段 | 名称 | 目标 | 对应任务文件 |
|------|------|------|-------------|
| 1 | 项目基础设施搭建 | pytest 框架 + 包结构修复 + CI 基础 | task01.md |
| 2 | 现有功能测试覆盖 | 为现有代码补充全面的单元测试和集成测试 | task02.md |
| 3 | 工艺配置系统重构 | YAML + Pydantic 配置体系，替代裸 Python 模块 | task03.md |
| 4 | 层定义动态化 | LayerStack 替代硬编码 layers.py，支持任意金属层数 | task04.md |
| 5 | 多 Track 支持 | 参数化单元高度和布线栅格，支持 7T/9T/12T 等 | task05.md |
| 6 | 多电源域与 BCD 基础 | 多 VDD/GND 支持，BCD 高压区域基础框架 | task06.md |
| 7 | 工艺迁移引擎 | 跨节点迁移工具（90nm→55nm 等） | task07.md |
| 8 | 脚本化二次开发接口 | Plugin/Hook 系统，SE 工程师友好的 Python API | task08.md |
| 9 | 端到端集成验证 | 多工艺节点完整流程验证 | task09.md |
| 10 | 生产级加固 | 性能优化 + 错误处理 + 日志 + 文档 | task10.md |

## 5. 技术约束与前提

### 5.1 环境要求

- Python 3.11+
- ngspice（用于时序表征）
- KLayout Python 模块（用于 GDS 操作和 LVS）
- Z3 求解器（用于 DRC 修正和 SMT 放置）

### 5.2 兼容性要求

- 现有 `dummy_tech.py` 必须能通过兼容层继续工作
- 现有 CLI 接口 `lclayout` / `lctime` / `sp2bool` 保持兼容
- 现有输出格式（GDS/LEF/MAG/Liberty）保持不变

### 5.3 不在范围内

- FinFET 工艺实现（仅保留架构扩展点）
- 模拟/RF 单元（仅数字标准单元）
- 全自动 PDK 导入（需手动编写工艺配置）
- 面向真实 Foundry 的 NDA 规则（使用开源 PDK 验证）

## 6. 风险评估

| 风险 | 影响 | 缓解措施 |
|------|------|----------|
| KLayout Python API 版本变化 | 高 | 抽象 KLayout 调用，测试中 mock |
| 重构引入回归 | 高 | **Task01/02 先建测试网，每次改动必过回归** |
| BCD LDMOS 模型复杂度 | 中 | 先实现 BCD 高压 CMOS 子集，再渐进扩展 |
| 性能退化（配置解析开销） | 低 | 配置对象运行时缓存，热路径 profiling |
| 依赖升级不兼容 | 中 | 放宽版本约束但锁定已验证版本（requirements.txt） |

## 7. 验收标准总结

**整体项目验收标准：**

1. 使用 YAML 工艺配置文件，成功生成 CMOS 180nm 标准单元库（至少含 INV/NAND2/NOR2/AND2/OR2/BUFX2/DFF），输出 GDS + LEF + Liberty
2. 同一配置修改 track 参数（7T→9T），重新生成单元库，GDS 合法
3. 使用另一套 YAML 配置（55nm），成功生成单元库
4. BCD 工艺配置下，成功生成包含高压单元的混合单元库
5. SE 工程师仅通过 YAML 配置 + Python 适配脚本（不修改核心代码），完成新工艺适配
6. 所有 pytest 测试通过，测试覆盖率 > 70%
